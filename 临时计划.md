1. 基础数据与通用功能
1.1 数据导入功能调试与修复
问题：在处理“港口类型”等枚举字段时，Excel导入的纯数字（Int32）无法赋给底层的枚举类型（Byte），导致导入失败。
临时方案：后端逻辑中，将捕获到的整数值强制转换为字符串，复用已有的字符串到枚举的转换逻辑。
结论：接受此临时方案以保证功能可用，远期可优化底层类型转换逻辑。
1.2 港口字典排序与数据唯一性
排序需求：前端港口字典下拉列表需按Code排序。
唯一性问题：发现海运和空运的港口编码（Code）大量重复，导致数据覆盖。
结论：港口的主键需变更为 `Code + 港口类型` 的复合主键。此项修改需与客户确认。
1.3 基础数据（字典）刷新性能问题
现象：进入某些页面（如新建账单）时，因频繁刷新字典（特别是港口信息，高达90+MB）导致明显卡顿。
原机制：每5分钟自动刷新一次。
决策：将自动刷新周期延长至 1小时。为需要即时更新的用户提供手动刷新按钮，或提示其重新登录以强制刷新。
2. 全局 UI/UX 优化与权限调整
2.1 权限设置页UI改造
需求：改造为左右分栏布局，左侧为可点击的目录树，右侧为对应的权限配置区，提升操作效率。
2.2 菜单结构调整（三级菜单）
需求：需支持三级菜单。二级菜单本身可点击进入页面，同时也可作为父节点展开/折叠三级子菜单。
信息架构建议：在「业务管理」下增设新分组「单据操作」，用于容纳主单/分单制作、舱单管理等新功能。
2.3 “查看所有申请单”权限逻辑调整
问题：权限（代号 1.3）被错误地应用在工作流（WF）接口上。
纠正：该权限应从WF接口移除，并应用到**获取费用申请单列表**的接口上。WF接口通过节点参与人隐式过滤，无需此权限。
3. 空运出口模块
3.1 核心功能：「主单领用登记」规划
背景：管理从航司领取的、具有唯一编码的主单（MAWB）。
业务逻辑：区分“登记”（直接从航司领）和“过单”（从其他代理借）。主单从上游“领入”，可自用或向下游“领出”。
数据模型：采用“主单台账 (MAWB_Stock) + 领入记录 (Stock_In) + 领出记录 (Stock_Out)”的三表结构。
MAWB规则：
结构：3位航司前缀 + "-" + 8位数字（末位为校验位，前7位 `mod 7` 得出）。
存储与显示：数据库统一存储无空格的标准化主单号。可增加`display_mawb`字段保存用户原始输入（如带空格的4-4格式）用于显示。
交互：领入为批量弹窗操作，领出为单张操作。在工作号中选择主单号时，从“未使用”的台账中选取。
3.2 功能：「主单/分单制作」规划
UI/UX优化:
长页面问题：主单编辑页面过长，操作按钮不便。解决方案为研究使用**内容区滚动容器**，或将“品名”、“集装器”等部分改���**弹窗编辑**。
通知提示位置：操作提示应在屏幕可视区中央显示。
计算逻辑:
委托明细：自动计算体积 (长*宽*高*件数)/1,000,000 和抛重 (长*宽*高*件数)/6000。明细的体积和重量需汇总至主表。
其他费用：根据付款方式(PP/CC)，将子表合计金额填入主表对应的PP或CC字段，另一字段置空。
分单特殊处理:
分单创建时，**不自动添加**默认的9个“其他费用”项，由用户手动添加。
4. 空运进口模块
4.1 新功能：「进口仓单」规划
背景：用于海关报关，数据源于出口方的主分单信息。主管单位为海关仓单科。
流程：系统接收/录入仓单数据，货物到港后操作员更新“运抵”、“理货”、“入库”等状态时间，并回传海关。本期先实现手动CRUD。
数据模型:
主单号格式：仓单系统使用**11位纯数字**主单号，与航空公司标准格式不同。系统需**原样记录**，在内部关联时再做格式化处理。
时间精度：各状态时间需精确到**分钟**。字段为空代表未发生。
主/分单识别：在子表中，通过“分单号”是否为空来区分。分单号为空的行是主单信息。
校验逻辑:
手动录入时，需校验同一主单下的所有分单件/重/体之和应等于主单行的数据。
所有行都必须有主单号。
5. 海运出口模块
5.1 新功能：「海运提单」规划 (主单/分单)
业务特点：仅涉及出口。周期长，单据通常独立于工作号早期创建（占单）。
关联关系：
主提单：一个工作号关联一个主提单。
分提单：一个工作号关联多个分提单。分提单与主提单之间通过共同的工作号建立逻辑联系，而非直接数据库外键。
字段与校验：
核心原则：所有字段默认为字符串、可为空，后端除明确指定外不作校验，以支持“先占单，后完善”的业务模式。
港口：使用港口全名，非三字代码。
交通工具：由“船名”和“航次”两个字段组成。
箱型箱量/件重体：数据逻辑参考“订舱”模块，可从工作号预填充，但用户仍可修改。
分提单：字段结构与主提单完全一致，可复用实体。其“提单编号”由货代自行生成。
6. 其他模块问题与需求
6.1 申请单批复后状态未刷新问题
问题：申请单批复完成后，前端页面状态未刷新，导致用户仍可编辑。需后端排查修复。
6.2 发票申请缺少手机号与Email
问题：在发票申请中选择购买方抬头后，未能自动带出其手机号和Email。需检查字段映射。
6.3 费用批量处理需求
痛点：大量费用需要逐一点击“新增”生成账单，操作重复。
需求：
确认“批量审批/审核费用”功能是否遗漏或被误删。
新增“**批量生成账单**”功能，一次性为所有未建账单的费用创建账单。
7. 后续规划与待办清单
7.1 下阶段重点：报关单模块
模块定位：作为数据准备和传输工具，而非全功能报关系统。
核心价值：利用系统内已有的基础数据（客户、商品等）帮助用户快速预填报关单，并能将数据传输至海关的“单一窗口”系统，解决在官方系统中重复手敲信息的痛点。
