
# 从零到一：用 Visual Studio + Web Deploy **自动发布网站到远程 IIS**（新手完全指南）

> **目的**：把你本地的 .NET/ASP.NET 网站，一键自动发布到远程 Windows 服务器的 IIS 网站上；发布过程自动完成“停止/释放文件 → 覆盖文件 → 恢复服务”，无需手动拷贝。
>
> **适用读者**：零基础/新手；只要能远程桌面登录服务器、能打开 Visual Studio 即可。
>
> **本文已去除无关的代码评估内容，仅保留并细化了“远程自动发布”相关步骤。**

## 一、发布所需前置条件
- 一台 **远程 Windows 服务器**（建议 Windows Server 2019/2022），可远程桌面登录。
- 服务器上已或将要安装 **IIS**（Web 服务器）。
- 本地开发环境：**Visual Studio 2022**（安装 ASP.NET/.NET 工作负载）。
- 你的网站项目（.NET Framework 或 .NET 6/7/8/… 都可）。

> **术语速览**  
> **Web Deploy（MSDeploy）**：微软官方的远程部署工具，走 HTTPS(8172) 到服务器的 **Web Management Service**（WMSvc）。

<a id="sec-2"></a>

## 二、在远程服务器上配置（只做一次）

<a id="sec-2-1"></a>

### 2.1 安装 IIS 及必要组件
1. 打开 **服务器管理器** → **添加角色和功能**。
2. 在 **Web 服务器(IIS)** 下，确保勾选：
   - **IIS 管理控制台**
   - **IIS 管理脚本和工具**（后续生成发布资料需要）
   - **管理服务**（WMSvc，Web Deploy Handler 依赖它）
   - **安全性 → 基本身份验证**（启用用户名/密码发布）
3. 完成向导后，重启 **IIS 管理器**（必要时重启服务器）。

<a id="sec-2-2"></a>

### 2.2 安装 Web Deploy 4.0（完整安装）

> **Web Deploy 不在 IIS 管理器中安装**，需要单独下载 MSI 安装包运行。微软官方提供了多语言版本。

1. 下载 Web Deploy 4.0（64 位）安装包（任选其一）：
   - **官方下载页**（可选择其他语言）：https://www.microsoft.com/en-us/download/details.aspx?id=106070
2. 运行安装，**安装类型选择 Complete / 完整（完整安装）**，确保包含：
   - IIS Deployment Handler
   - Web Management Service Agent
3. 安装完后在 **服务**（services.msc）检查：
   - **Web Deployment Agent Service**（MsDepSvc）→ 设为“自动”，并“正在运行”。
   - **Web Management Service**（WMSvc）→ 设为“自动”，并“正在运行”。

> **注意**：Web Deploy 4.0 的安装目录名可能仍为 `Microsoft Web Deploy V3\`（微软升级安装时复用旧目录名），这是正常现象，不需要修改注册表路径。

<a id="sec-2-3"></a>

### 2.3 开放防火墙/安全组端口 8172
- Windows 防火墙（管理员 PowerShell）：
  ```powershell
  netsh advfirewall firewall add rule name="Web Deploy 8172" dir=in action=allow protocol=TCP localport=8172
  ```
- 如果服务器在云厂商（阿里云/腾讯云/Azure 等），还要在 **安全组/网络ACL** 开放 **入站 TCP 8172**。

<a id="sec-2-4"></a>

### 2.4 启用 IIS 管理服务（WMSvc）并允许远程
1. 打开 **IIS 管理器**，点击左侧树的 **服务器根节点**。
2. 双击 **管理服务** → 勾选 **启用远程连接** → 右侧 **启动**（或先停止再启动）。

<a id="sec-2-5"></a>

### 2.5 为网站启用 Web Deploy 发布权限（生成 .publishsettings）
1. 在 **IIS 管理器** 左侧，右键你的 **网站**（例如 `MySite`）。
2. 选择 **部署(Deploy)** → **配置 Web 部署发布(Configure Web Deploy Publishing)**。
   - 若无“部署”菜单，返回 2.2 重新以 **Complete** 模式安装 Web Deploy。
3. 在弹窗中：
   - **允许发布的用户**：选择你登录服务器的本地管理员账户（如 `Administrator`）。
   - 点击 **Setup**。
4. 成功后界面会给出一个 **`.publishsettings` 文件**的保存位置（通常在当前用户桌面）。
5. 通过远程桌面把该文件 **拷回本地开发机**（拖拽/复制）。

> **如果此步骤反复报"无法授予用户完全控制权限"错误**：这是该 GUI 工具的已知问题，可以**直接跳过本步骤**。只要 MsDepSvc 和 WMSvc 两个服务正常运行、8172 端口已放行，Administrator 账户本身就有完整的部署权限，不依赖此 GUI 的额外授权。直接使用下面的方式 B 手动配置 pubxml 即可。

## 三、在开发机（VS）上配置与发布

<a id="sec-3-1"></a>

### 3.1 方式 A：导入 .publishsettings（最省心）
1. 在 **解决方案资源管理器** 右键你的 Web 项目 → **发布**。
2. **新建** → **导入配置文件(Import Profile)** → 选择刚才的 `.publishsettings` → **打开**。
3. VS 会自动填入服务器地址、网站名、用户；你只需 **输入密码并保存**。

> 建议把密码保存在项目的 `*.pubxml.user` 文件（VS 会自动管理）。注意本项目 `.gitignore` 已排除所有 `*.pubxml` 文件，因此 `.pubxml` 和 `.pubxml.user` **均不入库**，新成员需从团队获取或手动创建这两个文件。

<a id="sec-3-2"></a>

### 3.2 方式 B：手动编辑 `RemoteIIS.pubxml` 与 `RemoteIIS.pubxml.user`
> 适合已有发布配置文件、只需替换服务器参数的情况。

- **`Properties/PublishProfiles/RemoteIIS.pubxml`**（示例，以本项目 .NET 6 为例）：

<!-- [注释] 以下示例与实际 RemoteIIS.pubxml 存在差异：
  1. 实际配置中无 RuntimeIdentifier（框架依赖部署不需要指定）
  2. 实际配置中包含 StopRemoteAppPool / StartRemoteAppPool 两个 MSBuild Target，用于发布前后自动停止/启动远程应用程序池
  3. 实际配置中包含 MSDeployServiceURLBase 属性，避免 MSDeployPublish 篡改 URL 导致 site 参数重复
  4. 实际配置中包含 DeployAppPoolName 指定应用程序池名称
  请以实际 RemoteIIS.pubxml 文件为准
-->

```xml
<Project>
  <PropertyGroup>
    <WebPublishMethod>MSDeploy</WebPublishMethod>
    <LastUsedBuildConfiguration>Release</LastUsedBuildConfiguration>
    <LastUsedPlatform>Any CPU</LastUsedPlatform>
    <LaunchSiteAfterPublish>false</LaunchSiteAfterPublish>
    <ExcludeApp_Data>false</ExcludeApp_Data>
    <TargetFramework>net6.0</TargetFramework>
    <SelfContained>false</SelfContained>
    <!-- [注释] 实际配置中无此项，框架依赖部署不需要指定 RuntimeIdentifier -->
    <!-- <RuntimeIdentifier>win-x64</RuntimeIdentifier> -->
    <EnvironmentName>Production</EnvironmentName>
    <!-- ===== 请修改以下三项 ===== -->
    <MSDeployServiceURL>https://YOUR_SERVER_IP_OR_DOMAIN:8172/msdeploy.axd</MSDeployServiceURL>
    <DeployIisAppPath>YOUR_IIS_SITE_NAME</DeployIisAppPath>
    <UserName>YOUR_WINDOWS_USERNAME</UserName>
    <!-- ========================= -->
    <AuthType>Basic</AuthType>
    <SkipExtraFilesOnServer>false</SkipExtraFilesOnServer>
    <EnableMSDeployBackup>false</EnableMSDeployBackup>
    <AllowUntrustedCertificate>true</AllowUntrustedCertificate>
    <EnableMSDeployAppOffline>true</EnableMSDeployAppOffline>
  </PropertyGroup>
</Project>
```
- 把 `YOUR_SERVER_IP_OR_DOMAIN` 替换成服务器 **IP/域名**，`YOUR_IIS_SITE_NAME` 替换成 **IIS 网站名**，`YOUR_WINDOWS_USERNAME` 替换成有部署权限的 **Windows 用户名**。
- `AllowUntrustedCertificate=true` 允许自签名证书（测试环境方便）。
- `EnableMSDeployAppOffline=true` 启用维护页，自动释放文件锁（见下一章）。
- `SkipExtraFilesOnServer=false` 表示不跳过服务器上的多余文件（即会清理旧文件）。
- `SelfContained=false` 表示框架依赖部署，服务器需预装 .NET 6 Runtime。

- **`Properties/PublishProfiles/RemoteIIS.pubxml.user`**（保存密码，不入库）：
```xml
<Project>
  <PropertyGroup>
    <UserName>YOUR_WINDOWS_USERNAME</UserName>
    <_SavePWD>true</_SavePWD>
    <Password>你的服务器密码</Password>
  </PropertyGroup>
</Project>
```
- `<_SavePWD>true</_SavePWD>` 告知 VS 记住密码，避免每次发布时弹窗询问。

<a id="sec-3-3"></a>

### 3.3 一键发布（VS 界面 / 命令行两种）
- **VS 界面**：右键项目 → **发布** → 选择配置文件 → **发布** → 输出窗口出现 `Publish Succeeded` 即成功。
- **命令行（适合脚本/CI）**：在解决方案根目录打开 PowerShell：
```powershell
dotnet publish .\PowerLmsWebApi\PowerLmsWebApi.csproj `
  /p:PublishProfile=RemoteIIS `
  /p:Configuration=Release `
  /p:Password="你的服务器密码"
```
> 反引号 `` ` `` 是 PowerShell 的换行连接符；若用一行可去掉反引号。
> .NET 6+ 项目推荐使用 `dotnet publish` 替代传统 `msbuild`，前者自动处理 SDK 风格项目的还原与发布。

## 四、避免“文件被占用”——双重保障机制
多数失败的根因是：IIS 正在占用站点文件，导致覆盖失败。本项目采用**双重保障**机制：

**保障一：维护页（app_offline.htm）**
- 在 `*.pubxml` 中已经加了：
  ```xml
  <EnableMSDeployAppOffline>true</EnableMSDeployAppOffline>
  ```
- 发布时 MSDeploy 会在站点根目录临时放置 `app_offline.htm`，ASP.NET Core 检测到后释放文件锁；发布完自动删除，站点恢复。

**保障二：自动停止/启动应用程序池**
- 在 `RemoteIIS.pubxml` 中配置了 `StopRemoteAppPool`（发布前）和 `StartRemoteAppPool`（发布后）两个 MSBuild Target。
- 发布前通过 `msdeploy recycleApp` 停止远程应用程序池，等待 5 秒释放文件锁，发布完成后自动重启应用程序池。
- 这比仅使用维护页更可靠，能彻底释放所有文件句柄。

## 五、常见问题排查（贴问题给解法）
| 现象 | 可能原因 | 解决方法 |
|---|---|---|
| `Could not connect to the remote computer` | 8172 未放行/安全组拦截 | 检查 Windows 防火墙与云安全组，确保 **TCP 8172 入站**放行 |
| `(404) Not Found` | IIS Deployment Handler 未安装 | 以 **Complete** 方式重装 Web Deploy，确保含 *IIS Deployment Handler* |
| `(401) Unauthorized` | 认证失败/未启用基本身份验证 | 在 IIS 启用 **基本身份验证**；核对用户/密码；尝试 `Administrator` |
| `Web Management Service is not started` | WMSvc 未运行 | 在 **服务**或 **IIS 管理服务**面板启动 WMSvc，并设为 **自动** |
| 发布成功但文件未更新 | 被占用/未覆盖旧文件 | 打开 `EnableMSDeployAppOffline`；并确认 `SkipExtraFilesOnServer` 设为 `false`（不跳过清理服务器多余文件） |
| 证书错误 | 测试用自签名证书不受信 | 在 `pubxml` 设置 `AllowUntrustedCertificate=true`（仅测试环境使用） |

## 六、安全与最佳实践
- **最小化权限**：仅为需要发布的网站授予发布权限，避免给整个服务器放行。
- **发布配置不入库**：本项目 `.gitignore` 排除了所有 `*.pubxml` 文件，发布配置和密码均不入库。新开发者需手动创建或从团队获取 `RemoteIIS.pubxml` 和 `RemoteIIS.pubxml.user`。
- **先测再开公网**：若可行，优先内网/VPN 发布；公网发布务必限制源 IP。
- **备份配置**：`web.config`、站点重要文件在首次发布前做快照或备份。
